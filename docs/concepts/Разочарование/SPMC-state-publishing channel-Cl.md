# SPMC State Publishing Channel — Постановка задачи

`docs/contracts/SPMC-state-publishing-channel-task.md` · February 2026

---

## 1. Неформальное определение

Требуется передавать **текущее состояние** некоторого источника (writer) нескольким потребителям (readers) в системе реального времени таким образом, чтобы:

* каждый reader в любой момент мог получить **последнее опубликованное** состояние,
* ни один reader не получил **частично записанное** (torn) состояние,
* ни writer, ни reader не **блокировались** и не **ждали** друг друга,
* потребление памяти было **минимальным** относительно числа reader'ов.

---

## 2. Формальная постановка

### 2.1. Участники

* Ровно **один writer** `W` — периодически публикует новое состояние типа `T`.
* Ровно **N reader'ов** `R_0 .. R_{N-1}` — периодически читают последнее опубликованное состояние.

N фиксировано на этапе компиляции.

### 2.2. Тип данных

```
T: std::is_trivially_copyable<T>::value == true
```

Состояние — тривиально копируемый объект. Нет указателей на внешние ресурсы. Копирование = побайтовое.

### 2.3. Семантика передачи

**Latest-wins (снапшот).** Промежуточные публикации могут быть потеряны — reader'а не обязаны видеть каждую публикацию. Каждый reader при очередном опросе получает **наиболее свежее** из опубликованных состояний на момент начала чтения.

Это принципиально отличает задачу от очереди (queue), где каждый элемент должен быть доставлен.

### 2.4. Требования корректности

**R1. No torn read.**
Если reader получил значение `v` из публикации, то `v` является консистентной полной копией состояния `T` в том виде, в котором его записал writer. Частичное чтение (наблюдение состояния T в процессе записи) недопустимо.

**R2. No torn write.**
Запись writer'а не наблюдается reader'ом частично. Все байты `T` видимы атомарно относительно момента публикации. Формально: `slots[j] = value` happens-before `published.store(release)` happens-before `published.load(acquire)` у reader.

**R3. Validity.**
Если reader получил значение, оно было опубликовано writer'ом (не мусор, не неинициализированные данные после первой публикации).

### 2.5. Требования прогресса

**P1. Wait-free writer.**
`publish(value)` завершается за bounded число шагов независимо от состояния и действий любых reader'ов. Нет ожидания, нет retry, нет зависимости от скорости reader'ов.

**P2. Wait-free reader (при выполнении платформенных условий).**
`try_read(out)` завершается за bounded число шагов независимо от состояния и действий writer'а и других reader'ов. Нет ожидания, нет retry.

**P3. Отсутствие блокировок.**
Нет мьютексов, spinlock'ов, системных вызовов. Нет операций которые могут заблокировать поток на неограниченное время.

### 2.6. Требования к памяти

**M1.** Объём памяти на данные — `O(N) × sizeof(T)`. Конкретно: `(N+2) × sizeof(T)`.

**M2.** Нет динамического выделения памяти в `publish` и `try_read`. Все буферы аллоцируются при создании объекта.

### 2.7. Требования к WCET

**W1.** Worst-case execution time `publish()` и `try_read()` детерминирован и не зависит от числа reader'ов, частоты публикаций, и содержимого данных.

**W2.** WCET = `O(1)` атомарных операций + `O(sizeof(T))` на копирование.

## 3. Платформенные условия (Deployment Assumptions)

Требования P1–P3 и R1–R2 достижимы при выполнении одного из следующих условий развёртывания.

### DA-A. UP или одно RT-ядро без вытеснения (рекомендуется)

Writer и все reader'а выполняются на одном физическом ядре. Вытеснение в критических секциях отсутствует (`SCHED_FIFO`, `PREEMPT_RT` с отключённым preemption, bare-metal). Физического параллелизма нет — обе стороны строго wait-free, атомики нужны только как compiler barrier.

### DA-B. SMP с временны́м разделением

Writer завершает публикацию до начала нового тика опроса у reader'ов. Например: writer на выделенном RT-ядре, reader'а запускаются по таймеру с известным фазовым сдвигом относительно writer'а.

### DA-C. Произвольный SMP (ослабленные гарантии)

Writer и reader'а могут выполняться физически параллельно в любой момент без временны́х ограничений. В этом случае P2 (wait-free reader) **не достижимо** — reader становится lock-free с bounded retry (verify после установки claim). P1, P3, R1–R3 сохраняются.

---

## 4. Теоретический предел

### Теорема (недостижимость)

На произвольной SMP-системе (DA-C) **невозможно** одновременно выполнить P1, P2, R1 без примитива с consensus number ≥ 2.

**Доказательство (неформально):**
Reader должен атомарно выполнить `load(published)` и зарегистрировать claim на полученный слот. Эти два действия затрагивают **две независимые ячейки памяти** — атомарная операция над ними невозможна без CAS/LL-SC. Без атомарности существует окно в котором writer может выбрать слот для записи между `load` и `claim` у reader'а, нарушая R1. Закрыть окно можно только через retry у reader'а (нарушает P2) или через зависимость writer'а от reader'а (нарушает P1). ∎

Это частный случай результата Herlihy (1991) об иерархии консенсуса.

## 5. Что задача НЕ требует

* **Доставки каждой публикации** — reader'а не обязаны видеть все промежуточные состояния.
* **Упорядоченности между reader'ами** — разные reader'а могут видеть разные последние состояния.
* **Уведомления** — reader'а сами опрашивают канал в своём темпе (pull-модель).
* **Динамического числа участников** — N и K фиксированы при создании объекта.
* **Поддержки non-trivially-copyable типов** — указатели, виртуальные функции, RAII — вне области задачи.

## 6. Задержка свежести (Freshness Latency)

Задержка между публикацией writer'а и получением reader'ом:

```
Latency = T_w_period + T_r_period + T_hw_visibility
```

где:
* `T_w_period ∈ [0, T_w]` — reader пришёл в произвольный момент периода writer'а
* `T_r_period ∈ [0, T_r]` — writer опубликовал в произвольный момент периода reader'а
* `T_hw_visibility` — задержка cache coherency на SMP (0..50+ нс, платформенно-зависимо)

**Worst-case: `T_w + T_r`** — оптимум для pull-модели с независимыми периодами.

Verify-fail как дополнительная компонента задержки **отсутствует** при DA-A и DA-B.

## 7. Соответствие реализации

| Требование | Механизм в SPMCSnapshot |
|---|---|
| R1 No torn read | Инвариант W-NoOverwritePublished + claim протокол |
| R2 No torn write | `slots[j]=value` happens-before `published.store(release)` |
| R3 Validity | `initialized` атомик; `published` всегда валиден после первой публикации |
| P1 Wait-free writer | K=N+2 + Теорема slot availability; `ctz(candidates)` — O(1) |
| P2 Wait-free reader | Выполняется при DA-A и DA-B; lock-free при DA-C |
| P3 Нет блокировок | Только lock-free атомики: `load`, `store`, `fetch_or`, `fetch_and`, `fetch_add`, `fetch_sub` |
| M1 Память O(N) | K = N+2 слотов |
| M2 Нет аллокаций | Все буферы в теле объекта |
| W1 Детерминированный WCET | Фиксированное число операций, нет ветвлений на число итераций |
| W2 WCET O(1) | `ctz` + copy(T) + фиксированные атомики |
