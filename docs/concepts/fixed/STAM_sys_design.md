# STAM System Registry v1
## Принципы и граничные условия

---

## 1. Назначение реестра

Системный реестр — статическое описание системы исполнения RT/Non-RT задач и каналов связи.

Реестр фиксирует:
- набор задач
- набор каналов связи
- топологию исполнения
- параметры планирования
- compile-time гарантии корректности конфигурации

Реестр не является механизмом исполнения.

| Реестр | Runtime |
|--------|---------|
| описывает факты | исполняет задачи |
| проверяет инварианты | использует реестр как read-only таблицу |
| предоставляет данные runtime | — |

---

## 2. Основной принцип
### Реестр фиксирует факты и не принимает решений

Реестр не должен содержать механизмов вывода или оптимизации.

Реестр **не должен**:
- выбирать ядро исполнения
- вычислять приоритеты
- оптимизировать расписание
- балансировать нагрузку
- распределять каналы
- выводить зависимости

Реестр **только фиксирует**:
- что существует
- где исполняется
- как связано

---

## 3. Статичность

Реестр полностью статичен.

После старта системы:
- задачи не добавляются
- задачи не удаляются
- каналы не создаются
- каналы не удаляются
- параметры задач не меняются

Runtime имеет только read-only доступ.

---

## 4. Топология исполнения

### 4.1 Жёсткое закрепление задач за ядрами

Каждая задача должна иметь фиксированное ядро исполнения.

```
Task → CoreId
```

CoreId задаётся явно.

**Запрещено:**
- автоматическое распределение
- динамическое перемещение
- affinity masks
- "any core"

**Причина:**
- повторяемость поведения
- предсказуемость WCET
- конечность архитектуры

### 4.2 Ядра считаются независимыми исполнителями

Каждое ядро рассматривается как отдельная исполнительная среда:

```
Core i:
    Scheduler i
    Task set i
```

Взаимодействие между ядрами происходит только через каналы.

Топологический класс канала (`intra-core` / `inter-core`) является производным фактом:
он определяется из CoreId writer и readers на этапе компиляции и явного задания не требует.

## 5. Задачи

### 5.1 Задача — статический объект

Задача определяется:
- типом payload
- параметрами исполнения
- каналами ввода
- каналами вывода
- сигнальным битом (v1 constraint: производный, `signal_bit == task_index`)

Задача существует на этапе компиляции.

### 5.2 Тип задачи и сигнальный бит

#### Тип задачи — производный факт

Тип определяется наличием `period_ticks`:
- `period_ticks` задан → **Periodic**
- `period_ticks` не задан → **Signal-driven** (on-demand)

Задача имеет ровно один тип (`Periodic` или `Signal-driven`).

#### Сигнальный бит

Каждая задача имеет сигнальный бит.

```
signal_bit(task) == index_of(task) в реестре
```

Это локальное архитектурное ограничение v1 для структурной уникальности битов.
Явного объявления не требует.

Кто и когда устанавливает бит (ISR, IRQ-handler, другой код) — вне области реестра.
Сигнальный бит — инфраструктурный механизм внепланового запроса запуска, не поле типа задачи.

#### Periodic
Задача исполняется по таймеру:
- `period_ticks ≥ 1`

Сигнальный бит может быть установлен для дополнительного запуска вне периода.

#### Signal-driven (on-demand)
Задача исполняется только по решению шедулера.
Сигнальный бит — единственный механизм активации.
`period_ticks` отсутствует.

Задачи не взаимодействуют напрямую. Шедулер является единственным механизмом активации.

### 5.3 Приоритеты

Каждая задача имеет статический приоритет.

```
uint8_t priority
```

Реестр **не должен**:
- вычислять приоритеты
- изменять приоритеты
- оптимизировать приоритеты

Порядок задач с одинаковым приоритетом фиксирован.
Tiebreaker в v1: порядок объявления задач в реестре.

### 5.4 Таймаут наблюдаемости

Каждая задача имеет явный атрибут:
```
observable: bool
```

Если `observable == true`, задача обязана иметь `timeout_ticks`.

Timeout используется системой safety.

Timeout задаётся явно.

Реестр не вычисляет timeout автоматически.

## 6. Каналы связи

### 6.1 Канал — статический объект

Канал определяется:
- типом (`EventChannel<T>` / `StateChannel<T>`)
- типом данных `T`
- писателем
- количеством читателей

Канал существует на этапе компиляции.

### 6.2 Ownership

Ownership зависит от типа канала:
```
EventChannel<T>:  WriterCount == 1, ReaderCount == 1
StateChannel<T, N>:  WriterCount == 1, ReaderCount == N
```

Writer всегда ровно один. N — точное число reader-задач, задаётся при объявлении канала. Это compile-time инварианты.

### 6.3 Типы каналов

#### EventChannel\<T\>
- Протоколирование событий: payload типа `T`
- Не влияет на сигнальную маску; не является механизмом активации задач
- Семантика: FIFO (один writer, один reader)
- Используется для записи событий с типизированными данными
- Примеры реализаций: `SPSC Ring`

#### StateChannel\<T\>
- Несёт текущее состояние: payload типа `T`
- Не влияет на сигнальную маску; reader читает последнее значение при плановом запуске
- Семантика: SPMC Snapshot (один writer, несколько readers)
- Используется periodic задачами для чтения данных
- Примеры реализаций: `SPMC Snapshot`

Тип канала задаётся при объявлении и не изменяется runtime.

### 6.4 Связи фиксированы

Связи между задачами и каналами заданы статически.

```
Task → Channel → Task
```

**Запрещено:**
- динамическое подключение
- подписки
- pub/sub
- discovery

## 7. Compile-time проверки

Реестр должен позволять compile-time проверку конфигурации.

### 7.1 Ownership каналов
```
EventChannel<T>:  WriterCount == 1, ReaderCount == 1
StateChannel<T, N>:  WriterCount == 1, ReaderCount == N
```

### 7.2 Типовая корректность каналов
```
EventChannel<T>:  читается как T, пишется как T
StateChannel<T>:  читается как T, пишется как T
```

### 7.3 CoreId корректен
```
CoreId < NUM_CORES
```

### 7.4 Параметры Periodic задач
```
period_ticks ≥ 1
```

### 7.5 Timeout корректен

```
if task.observable == true → timeout_ticks must be defined
```

### 7.6 Сигнальный бит
```
v1 constraint: каждая задача имеет сигнальный бит: signal_bit == task_index
все биты уникальны (гарантировано структурно через task_index)
```

### 7.7 Writer канала существует
```
для каждого канала: writer task объявлена в реестре
```

### 7.8 Readers каналов существуют
```
для каждого reader в канале: reader task объявлена в реестре
```

### 7.9 Tiebreaker приоритетов
```
для задач с одинаковым приоритетом:
порядок детерминирован и равен порядку объявления в реестре
```

### 7.10 Ёмкость сигнальной маски
```
task_count <= SIGNAL_MASK_WIDTH
```

---

## 8. Ограничение модели времени

Единица времени реестра — **tick**.

Реестр использует только ticks:
- `period_ticks`
- `timeout_ticks`

Реестр **не использует**:
- milliseconds
- microseconds
- floating point time

Перевод времени выполняется вне реестра.

---

## 9. Границы ответственности реестра

**Реестр отвечает за:**
- описание задач
- описание каналов
- описание топологии
- compile-time инварианты

**Реестр не отвечает за:**
- выполнение задач
- переключение контекста
- scheduling алгоритмы
- WCET вычисление
- балансировку нагрузки
- оптимизацию

---

## 10. Ограничение расширяемости

Реестр v1 намеренно ограничен.

В v1 **отсутствуют**:
- автоматическое распределение задач по ядрам
- динамические задачи
- динамические каналы
- runtime конфигурация
- dependency graphs
- scheduling policies
- affinity masks
- resource models
- utilization analysis

Добавление возможностей допускается только через v2.

---

## 11. Архитектурная цель

Реестр должен оставаться:
- конечным
- анализируемым человеком
- полностью обозримым
- пригодным для формального анализа

Человек должен быть способен:
- перечислить все задачи
- перечислить все каналы
- понять топологию
- понять статическую приоритетную структуру

**без инструментов.**

---

## 12. Критерий завершённости модели

Модель реестра считается завершённой если:
- можно описать систему исполнения
- runtime может работать только на данных реестра
- compile-time проверки закрывают конфигурационные ошибки
- новые задачи можно добавлять без изменения **схемы** реестра
- добавление нового типа триггера требует изменения схемы и выполняется только в v2
