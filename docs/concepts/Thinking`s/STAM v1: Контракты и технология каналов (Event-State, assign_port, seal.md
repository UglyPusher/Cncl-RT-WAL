## Видение задачи

Задача — спроектировать **каналы связи** как самостоятельную архитектурную единицу.
Из уже зафиксированного (registry v1) известно что: каналы бывают двух типов, у каждого по одному writer-у, топология статична. Это верхний уровень. Теперь нужно спуститься вниз — до API, до памяти, до RT-контракта.

Я вижу три слоя, которые нужно закрыть:

1. **Внешний слой** — как payload-код *использует* порт (что пишет, что читает, какой тип возврата)
2. **Средний слой** — что такое `InPort<T>` / `OutPort<T>` как объекты (где живут, чем владеют, как привязаны к каналу)
3. **Нижний слой** — как реализован сам канал (SPSC Ring / SPMC Snapshot, memory ordering, inter-core)

Ниже — соображения по каждому слою и ключевые вопросы.

---

### Слой 1: API использования порта

Payload получает доступ к своим портам и делает `read` / `write` внутри `step(tick_t now)`.

Предположительная форма (требует решения):

```cpp
// writer-задача:
out_port.write(value);          // OutPort<T>

// reader-задача:
bool ok = in_port.read(val);    // EventChannel — есть/нет данных
T snapshot = in_port.get();     // StateChannel — последнее значение
```

**Открытый вопрос A**: Как payload получает ссылку на порт?

Варианты:
- Порт как поле payload-а (`SensorPort out_ { registry.port<OUT_SENSOR>() };`)
- Порт передаётся в конструктор payload-а
- Порт доступен через дескриптор задачи

Выбор здесь влияет на всю технологию работы разработчика.

---

### Слой 2: Природа Port-объектов

`InPort<T>` / `OutPort<T>` — это не буферы. Это **дескрипторы доступа** (ручки/handles), указывающие на хранилище в канале.

Предположительная модель:
- Хранилище (ring / snapshot) живёт **в канале** как статический объект
- Порт содержит указатель/ссылку на это хранилище — присваивается при `channel.bind_in/out()`
- После `seal()` порт заморожен: указатель валиден на всё время жизни системы

Это симметрично модели `hb_ref` в TaskWrapper — тот же паттерн двухфазной инициализации.

**Открытый вопрос B**: Порт — value-объект или reference-wrapper?

Если value — хранит T-шный указатель внутри себя.
Если reference-wrapper — это просто `T* ptr` с методами. Скорее второе (нет heap, нет копирования буфера).

### Слой 3: Реализация канала

**EventChannel<T>** → SPSC Ring:
- Фиксированная ёмкость (capacity — compile-time параметр?)
- lock-free push/pop
- **Открытый вопрос C**: что делать при переполнении ring?
  - `write()` возвращает `bool` (false = dropped) — информирует writer
  - `write()` overwrite oldest — silent loss, проще для RT
  - `write()` ничего не делает — тихо дропает
  - Нужен consensus: drop-policy это часть контракта типа или параметр канала?

**StateChannel<T>** → SPMC Snapshot:
- В репо уже есть `SPMCSnapshot` (primitives, сессии 0033–... прошлого цикла)
- Семантика: reader всегда видит последнее опубликованное значение
- **Открытый вопрос D**: начальное значение — кто отвечает?
  - `T{}` default-construct при инициализации канала?
  - Явное `initial_value` в дескрипторе канала?
  - Неопределено до первого write (опасно для RT)?

### Intra-core vs Inter-core

Топологический класс — производный (из CoreId writer и readers).

**Открытый вопрос E**: нужны ли разные имплементации для intra vs inter?

На Cortex-M4 MCU с общей памятью, но разными кэшами:
- SPSC Ring inter-core требует `memory_order_acquire/release` на head/tail
- SPSC Ring intra-core — достаточно relaxed (если один планировщик/прерывание)?
- SPMCSnapshot уже использует acquire/release — межъядерно безопасен по построению

Возможный подход: единая реализация с корректными memory_order-ами — проще и надёжнее, чем две реализации.

### Жизненный цикл канала

По аналогии с wrapper:

```
[создание: Channel<T> ch]                  ← хранилище statically allocated
[bind: ch.bind_in(task.out_port)]          ← порт writer-а привязан к каналу
[bind: ch.bind_out(task.in_port)]          ← порт reader-а привязан к каналу
[seal()]                                   ← проверка инвариантов, freeze
[RT: out_port.write(v) / in_port.read()]   ← задачи работают через порты
```

Никакого heap. Никаких виртуальных функций. Вся конфигурация до seal().

### Что прояснилось из примитивов

Оба примитива уже **разделены на Core + Writer + Reader**:

```
SPSCRing<T, N>
    .writer() → SPSCRingWriter<T,N>   — хранит ссылку на Core, метод push()
    .reader() → SPSCRingReader<T,N>   — хранит ссылку на Core, метод pop()

SPMCSnapshot<T, N>
    .writer() → SPMCSnapshotWriter<T,N>  — метод publish()
    .reader() → SPMCSnapshotReader<T,N>  — метод try_read()
```

Writer/Reader — это **лёгкие handle-объекты** (хранят только ссылку на Core). Именно они и есть «интерфейсы на запись/чтение». Каналу остаётся быть обёрткой над примитивом — ровно как TaskWrapper над payload.

---

### Простая модель (без лишних слоёв)

```
EventChannel<T, Capacity>
    хранит: SPSCRing<T, Capacity>   // статическое хранилище, zero heap
    даёт:   .writer() → SPSCRingWriter<T,Capacity>   → идёт в payload writer-задачи
            .reader() → SPSCRingReader<T,Capacity>   → идёт в payload reader-задачи

StateChannel<T, N>
    хранит: SPMCSnapshot<T, N>
    даёт:   .writer() → SPMCSnapshotWriter<T,N>
            .reader() → SPMCSnapshotReader<T,N>
```

`InPort<T>` / `OutPort<T>` из реестрового дескриптора — это и есть эти Writer/Reader объекты. Никакого промежуточного слоя нет.

### Вопрос A — остаётся, но переформулирован

Из примитивов видно: Writer/Reader — movable handle (move = transfer of role). Payload может хранить их как поля.

Конкретный вопрос: **кто вызывает `channel.writer()` / `channel.reader()` и как результат попадает в payload?**

class SensorTask {

class SensorTask {

static SensorTask sensor_task { sensor_channel.writer() };

### Вопрос D — возможно снят примитивом

`SPMCSnapshot.try_read()` возвращает `false` до первого `publish()` — это уже встроено в примитив через флаг `initialized`. Reader-задача обязана обрабатывать `false`.

Если reader обрабатывает `false` (= нет данных ещё), явный `initial_value` не нужен. Это уже семантика примитива: «нет данных до первого publish».

**Уточняющий вопрос**: допустимо ли для периодической reader-задачи получить `false` на первом тике? Если да — D закрыт самим примитивом. Если нет (надо гарантировать данные к первому тику) — нужен явный `initial_value` через `publish()` при init().

### C — уточнён, не снят

Важное уточнение от оператора: drop-policy — это выбор **примитива**, а не API порта. Есть два кольцевых примитива:
- **last-loss** (новое теряется при переполнении)
- **first-loss** (старое вытесняется при переполнении)

Порт предоставляет только интерфейс read/write — обработка ситуации (что делать с флагом overflow) остаётся на стороне задачи внутри `step()`.

**Остаётся вопрос**: как разработчик выбирает примитив при объявлении EventChannel? Варианты:

- `EventChannel<T, DropPolicy::LastLoss, Capacity>` — параметр типа
- `EventChannel<T, LastLossRing<T, N>>` — явная передача примитива
- два отдельных типа `EventChannelLL<T,N>` / `EventChannelFL<T,N>` — без шаблонного параметра

Предлагаю: `EventChannel<T, Capacity, DropPolicy>` — compile-time параметры, enum DropPolicy.
Это читаемо и не усложняет тип порта: `InPort<T>` и `OutPort<T>` остаются одинаковыми независимо от политики.

### A — предложение с аргументами

Оператор описывает паттерн: у payload "торчит наружу" структура/слот, куда реестр/bind "запихивает" reader/writer интерфейс.

Предлагаю: **Dependency Injection через конструктор** — payload получает ссылку на порт как аргумент конструктора.

```cpp
// OutPort<T> и InPort<T> объявляются рядом с TaskDescriptor, не внутри payload-а
static OutPort<SensorData> sensor_out;   // "торчит наружу" — виден реестру
static SensorTask          sensor_task { sensor_out };  // DI
static TaskWrapper         sensor_wrapper { sensor_task };

// в payload:
class SensorTask {
    OutPort<SensorData>& out_;
public:
    explicit SensorTask(OutPort<SensorData>& out) noexcept : out_(out) {}
    void step(tick_t now) noexcept { out_.write(make_reading()); }
};
```

Аргументы за:
1. **Payload изолирован от реестра** — не импортирует registry, не знает о seal(), не зависит от имён
2. **Тестируемость** — тест подаёт mock-порт в конструктор и проверяет payload без реестра
3. **Статическая типизация** — нет поиска по имени в runtime, нет reflect
4. **"Торчит наружу"** — OutPort/InPort объявлены в том же файле что и TaskDescriptor, видны для bind; payload их не создаёт и не владеет ими
5. **Симметрия с hb_** — паттерн двухфазной инициализации: OutPort создан без канала, bind() подключает хранилище

Альтернатива — порт как поле payload-а:
```cpp
class SensorTask {
    OutPort<SensorData> out_;  // поле внутри payload
    ...
};
```
Проблема: реестр должен "видеть" это поле для bind() — либо через указатель-на-член, либо через метод `get_port()`. Это протекание реестра в payload. Отвергаю.

### B — предложение с аргументами

`InPort<T>` / `OutPort<T>` — **reference-wrapper**: хранит указатель на хранилище канала, методы read/write делегируют в примитив.

```cpp
template<typename T>
class OutPort {
    ChannelWriter<T>* impl_ = nullptr;   // указатель на интерфейс примитива
public:
    // вызывается реестром при bind_in()
    void attach(ChannelWriter<T>& impl) noexcept { impl_ = &impl; }

    void write(const T& v) noexcept {
        assert(impl_);   // debug: порт не привязан
        impl_->push(v);
    }

    // не копируется — аналогично TaskWrapper
    OutPort(const OutPort&) = delete;
    OutPort& operator=(const OutPort&) = delete;
};
```

Аргументы за reference-wrapper vs value:
- Нет копирования хранилища — только указатель, 8 байт на 64-bit
- Двухфазная инициализация: создан без канала (`impl_ = nullptr`), bind() подключает — тот же паттерн что `hb_` в TaskWrapper
- Не копируется — порт уникален, нет риска двух writer-ов через копию
- В debug — assert что attach() был вызван до write/read

`ChannelWriter<T>` и `ChannelReader<T>` — это интерфейсы конкретного примитива (SPSC ring, SPMC snapshot writer/reader-стороны). Они живут внутри объекта канала. Порт хранит только указатель на них.

### D — предложение с аргументами

**Явный `initial_value` обязателен для StateChannel.**

Аргументы:

| Вариант | За | Против |
|---|---|---|
| `T{}` неявно | Проще API | "Магия"; нулевой датчик давления = корректное состояние? Нет |
| Явный `initial_value` | Разработчик думает о начальном состоянии; safety-explicit | Чуть больше кода |
| Undefined до первого write | — | Недопустимо: reader не знает валидны ли данные |

В STAM принят принцип явности: `task_name` обязателен, `core_id` задаётся явно, `timeout_ticks` задаётся явно если `observable`. StateChannel следует тому же принципу.

Предлагаю:
```cpp
StateChannel<SensorData, initial_value>  // шаблонный параметр (если T — POD)
// или
StateChannel<SensorData> ch { SensorData{} };  // конструктор с initial_value
```

Для compile-time известных значений — шаблонный параметр. Для runtime-инициализируемых — через конструктор до `seal()`. Оба варианта явны.

**Следствие**: SPMCSnapshot при создании инициализируется переданным значением, reader никогда не читает "мусор".

### Ключевой открытый вопрос: SPMC reader multiplicity

Оператор: "у SPMC исходящих каналов не один". Это корень незакрытого пункта 2.

Из примитивов:
```
SPMCSnapshot<T, N>:   N — compile-time, максимум N независимых reader-handles
  .reader()  →  SPMCSnapshotReader<T,N>    // можно вызвать до N раз
  .writer()  →  SPMCSnapshotWriter<T,N>    // ровно 1 раз

Mailbox2Slot (SPSC snapshot): ровно 1 reader, ровно 1 writer
```

Для StateChannel с ReaderCount ≥ 1: нужен SPMCSnapshot, N задаётся при объявлении канала. Каждый из N reader-tasks при `assign_port` получает свой независимый handle через отдельный вызов `channel.reader()`.

**Проблема**: `StateChannel<T>` без N в типе — невозможно создать `SPMCSnapshot<T, N>` без N. Значит тип StateChannel должен нести N:

```
StateChannel<T, N>    // N = максимальное число reader-ов (compile-time)
    хранит: SPMCSnapshot<T, N>
    .writer()          → SPMCSnapshotWriter<T,N>   // ровно 1 вызов
    .reader()          → SPMCSnapshotReader<T,N>   // до N вызовов, каждый независим
```

**Последствие для assign_port**: за StateChannel<T,N> можно сделать assign_port для reader не более N раз. seal() проверяет что число reader-assign ≤ N. Превышение = ошибка конфигурации.

---

### Схема assign_port для StateChannel (3 readers):

```cpp
static StateChannel<SensorData, 3> sensor_ch;  // N=3

// в bootstrap:
display_task.assign_port("in_sensor",  sensor_ch, reader);  // reader #1
logger_task.assign_port ("in_sensor",  sensor_ch, reader);  // reader #2
control_task.assign_port("in_sensor",  sensor_ch, reader);  // reader #3
sensor_task.assign_port ("out_sensor", sensor_ch, writer);  // writer

// seal() проверяет: readers == 3 == N, writer == 1
```

Каждый вызов `sensor_ch.reader()` внутри assign_port возвращает отдельный `SPMCSnapshotReader` — это и есть "исходящих каналов не один".

### Единственный открытый вопрос: `StateChannel<T, N>` — N в типе?

Проблема: `SPMCSnapshot<T, N>` требует N на этапе компиляции. Без N создать бэкенд нельзя.

**Предлагаю: `StateChannel<T, MaxReaders>`** — N как capacity bound:

```
StateChannel<T, MaxReaders>
    хранит: SPMCSnapshot<T, MaxReaders>
    .writer()  → ровно 1 вызов, 1 writer handle
    .reader()  → до MaxReaders независимых вызовов, каждый → свой reader handle
```

Разница с "N = точное число readers":
- `StateChannel<T, 4>` с 2 реальными reader-задачами — допустимо (2 ≤ 4)
- `seal()` проверяет: `readers_assigned ≤ MaxReaders`, а не `== MaxReaders`
- Это аналогично `Capacity` в SPSCRing — указываем bound, не точное значение

**Assign_port для N readers:**

```cpp
static StateChannel<SensorData, 4> sensor_ch;  // max 4 readers

display_task.assign_port("in",  sensor_ch, reader);  // handle #1
logger_task.assign_port ("in",  sensor_ch, reader);  // handle #2
// итого 2 reader-assign < 4 = OK

sensor_task.assign_port ("out", sensor_ch, writer);  // writer

// seal(): 2 readers ≤ 4 MaxReaders ✓, 1 writer ✓
```

Каждый вызов `sensor_ch.reader()` внутри assign_port возвращает отдельный `SPMCSnapshotReader` — они независимы, указывают на один `SPMCSnapshotCore`.

---

### Почему не `StateChannel<T>` без N?

Нет возможности создать `SPMCSnapshot<T, N>` без знания N на этапе компиляции. Хранить N отдельным runtime-числом означает: либо лимитировать MAX глобально (`static SPMCSnapshotCore<T, GLOBAL_MAX>`), либо иметь heap. Оба варианта хуже явного параметра.

### Почему оператор прав

Я предложил "capacity bound": `StateChannel<T, 4>` с 2 readers — допустимо. Это удобно, но создаёт два класса проблем:

1. **Молчаливый перерасход памяти**: SPMCSnapshot<T, 4> выделяет K=6 слотов. Если реальных readers 2 — 4 слота выделено зря. На embedded это болезненно.
2. **Молчаливое забывание**: разработчик объявил 4, привязал 2, оставил 2 зависших слота. Система стартует, работает — ошибка не видна. Это хуже, чем перерасход.

seal() — единственное место где это можно поймать. Если не квакнуть здесь — ошибка уйдёт в production.

---

### Исправленная модель: N = точное число readers

```
StateChannel<T, N>
    N = ТОЧНОЕ число reader-задач (не максимум)
    seal() проверяет: readers_assigned == N  (не ≤ N)
    seal() проверяет: writer_assigned == 1
```

Нарушение любого из условий → ошибка в `SealResult` с указанием канала и проблемы.

Это уже частично есть в `sys_registry_structure.md`:
```
| Ownership StateChannel | WriterCount == 1, ReaderCount ≥ 1 |
```
Нужно уточнить: ReaderCount при seal() должен точно совпадать с N из типа.

---

### Следствие для разработчика

```cpp
// Правильно: объявляю ровно столько readers сколько задач читают
static StateChannel<SensorData, 2> sensor_ch;  // 2 задачи будут читать

display_task.assign_port("in",  sensor_ch, reader);  // #1
logger_task.assign_port ("in",  sensor_ch, reader);  // #2

// seal(): 2 == N ✓

// Неправильно — поймает seal():
static StateChannel<SensorData, 4> sensor_ch;  // объявил 4
display_task.assign_port("in",  sensor_ch, reader);  // привязал 1
// seal(): 1 ≠ 4 → SealError: "sensor_ch: 3 reader ports unbound"
```

---

### Обобщение: seal() проверяет ВСЕ порты всех каналов

Это же правило применяется и к EventChannel:

| Канал | Writer bound | Readers bound |
|---|---|---|
| `EventChannel<T,C,P>` | == 1 | == 1 |
| `StateChannel<T,N>` | == 1 | == N |

Ни одного непривязанного порта — иначе SealError.

---

### Статус StateChannel<T,N>

Вопрос закрывается:
- N в типе = точное число readers (не bound)
- seal() enforce == N, не ≤ N
- Это защищает от забытых привязок и перерасхода памяти

Добавить в `fixed.md` как **[FIXED-StateChannel-N]**: N = точное число readers; seal() требует полной привязки всех портов.
