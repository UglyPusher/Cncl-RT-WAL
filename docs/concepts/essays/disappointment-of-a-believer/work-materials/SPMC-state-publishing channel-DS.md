# SPMC-state-publishing channel: Problem Definition and Requirements

## 1. Назначение

Примитив **SPMC-state-publishing channel** (Single Publisher, Multiple Consumers — state snapshot) предназначен для организации **безопасной передачи снапшотов состояния** от одного потока-писателя (writer) к фиксированному множеству потоков-читателей (readers) с семантикой **latest-wins** (т.е. промежуточные публикации могут быть потеряны).

Канал спроектирован для использования в **real-time системах** с ограниченной памятью, где требуется детерминированное время выполнения (wait-free, bounded WCET) и отсутствие блокировок.

---

## 2. Класс решаемых задач

Примитив решает задачу **безопасной публикации разделяемого состояния** в условиях:

- **Ограниченная память** — требуется минимизировать число копий данных при большом количестве читателей.
- **SMP-системы** — корректен на многопроцессорных платформах с учётом модели памяти (C++11 atomics).
- **Вытеснение** — должен сохранять корректность при вытеснении любого потока планировщиком.
- **Real-time** — все операции должны быть wait-free и иметь предсказуемое верхнее время выполнения (WCET).
- **Фиксированное число читателей** — N задаётся на этапе компиляции.

---

## 3. Формальная постановка задачи

### 3.1 Дано

- Тип данных `T`, удовлетворяющий `std::is_trivially_copyable<T>::value == true`.
- Число читателей `N` (константа времени компиляции, `N >= 1`).
- Число слотов `K = N + 2`.
- Один поток-писатель (writer).
- `N` потоков-читателей (readers), каждый с идентификатором от 0 до N-1.
- Аппаратная платформа: SMP с когерентными кэшами, поддерживающая атомарные операции с барьерами памяти (C++11 memory model).

### 3.2 Требуется реализовать примитив, предоставляющий:

- **Метод `publish(value: T)`** (вызывается только writer'ом):
  - Атомарно публикует новое состояние `value`.
  - Гарантирует, что после завершения метода все readers, которые начнут чтение, увидят либо это состояние, либо более позднее.
  - Не блокируется и не зависит от действий readers (wait-free).

- **Метод `try_read(out: &T) -> bool`** (вызывается любым reader'ом):
  - Если хотя бы одна публикация уже состоялась, копирует текущее опубликованное состояние в `out` и возвращает `true`.
  - До первой публикации возвращает `false`.
  - Не блокируется и не зависит от действий writer'а или других readers (wait-free в смысле bounded steps, при соблюдении условий эксплуатации — см. п. 5).

### 3.3 Дополнительные условия

- Память под данные: ровно `K` слотов типа `T`, плюс служебные атомарные переменные.
- Никаких динамических выделений памяти, системных вызовов, блокировок (spin-lock, mutex).
- Возможность компиляции для встраиваемых систем с отключёнными исключениями и RTTI.

---

## 4. Желаемые свойства (гарантии)

### 4.1 Safety

1. **Консистентность снапшота**  
   Если `try_read(out)` возвращает `true`, то `out` содержит полную и непротиворечивую копию состояния, которое было опубликовано writer'ом в некоторый момент времени (не ранее момента вызова `publish` и не позднее момента завершения `try_read`).

2. **Отсутствие torn reads**  
   Ни при каких обстоятельствах reader не может прочитать частично обновлённые данные (пересечение записи writer'а и чтения reader'а в одном слоте).

3. **Отсутствие состояния NONE**  
   После первой успешной публикации `published` всегда указывает на валидный слот. Нет «окна недоступности» данных.

4. **Latest-wins**  
   Если writer публикует несколько значений подряд, reader может увидеть только последнее из них (промежуточные могут быть потеряны).

### 4.2 Progress

1. **Writer — wait-free**  
   Количество шагов, выполняемых writer'ом при публикации, ограничено сверху константой, не зависящей от числа читателей и их действий. (При условии, что выбор свободного слота выполняется за O(1), например через `ctz`.)

2. **Reader — wait-free (в смысле bounded steps)**  
   Количество шагов, выполняемых reader'ом при успешном или неуспешном чтении, ограничено сверху константой, не зависящей от действий writer'а и других readers.  
   *Примечание:* В общем случае на SMP без дополнительных предположений wait-free для reader'а недостижим — требуется verify, что приводит к bounded retry (lock-free). Поэтому в спецификации должно быть явно указано, при каких условиях reader остаётся wait-free.

### 4.3 Resource bounds

- Память: `(N+2) * sizeof(T) + O(1)` байт для служебных структур.
- Время: WCET для `publish` и `try_read` должно быть вычислимо и не зависеть от N (кроме выбора слота, который может быть линейным по K, но K фиксировано, поэтому bounded).

---

## 5. Условия корректности на SMP

Как показано в теоретическом анализе, для обеспечения wait-free прогресса reader'а без verify необходимы дополнительные предположения о среде исполнения:

- **Условие A (UP или закрепление за одним ядром):**  
  Writer и все readers выполняются на одном и том же ядре, либо система однопроцессорная, и вытеснение в критическом окне запрещено (preemption disabled между `published.load()` и установкой `busy_mask`).

- **Условие B (временное разделение):**  
  Writer и readers работают на разных ядрах, но гарантируется, что writer не может выполнить более одной публикации за время между последовательными операциями reader'а (например, из-за фазовых сдвигов или приоритетов).

- **Условие C (использование verify):**  
  Если ни одно из условий A или B не выполняется, для безопасности необходима версия с verify (как в v5), при которой reader становится lock-free (bounded retry), а writer остаётся wait-free.

В документе должно быть явно указано, какой вариант предполагается, и что псевдокод без verify применим только при соблюдении условий A или B.

---

## 6. Ключевые инварианты (для реализации с verify-free)

Для варианта без verify (условия A или B) необходимо поддерживать следующие инварианты:

- **W-NoOverwritePublished:** Writer никогда не пишет в слот, который является текущим опубликованным (`published`).
- **Claim before read:** Reader перед чтением слота обязательно устанавливает бит занятости (`busy_mask`) и увеличивает счётчик (`refcnt`).
- **Busy отражает refcnt:** `busy_mask[i] == 1` только если `refcnt[i] > 0` (консервативно, с учётом порядка операций).
- **Published стабилен:** После того как reader захватил слот по индексу из `published`, этот слот не может быть выбран writer'ом до тех пор, пока reader не снимет захват (благодаря комбинации `published` и `busy_mask`).

---

## 7. Границы применимости

- N фиксировано на этапе компиляции и не может быть изменено без перекомпиляции.
- Максимальное число читателей ограничено разрядностью масок (`uint32_t` → N ≤ 30, `uint64_t` → N ≤ 62) и типом `refcnt` (`uint8_t` → N ≤ 254).
- Тип T должен быть тривиально копируемым (POD).
- Платформа должна поддерживать lock-free атомарные операции для используемых типов (`std::atomic` с гарантией `is_always_lock_free`).

## 8. Сравнение с альтернативами

| Примитив | Память | Writer | Reader | Torn read |
|----------|--------|--------|--------|-----------|
| N × Mailbox2Slot | 2N × sizeof(T) | wait-free, O(N) | wait-free, O(1) | нет |
| SeqLock | 1 × sizeof(T) | wait-free, O(1) | lock-free (retry) | нет |
| SPMCSnapshot (v6.2) | (N+2) × sizeof(T) | wait-free, O(1) | wait-free (при усл.) / lock-free (общ.) | структурно (при усл.) |

---

## 9. Заключение

Задача SPMC-state-publishing channel формулируется как построение примитива синхронизации с минимальным потреблением памяти, wait-free прогрессом для писателя и (в идеале) для читателей, при гарантии отсутствия torn reads. Решение достигается за счёт использования `N+2` слотов, атомарного указателя `published` и масок занятости. На SMP без дополнительных предположений требуется verify, что делает читателя lock-free; в контролируемых условиях (UP, закрепление за ядром, временное разделение) можно сохранить wait-free для обеих сторон.
