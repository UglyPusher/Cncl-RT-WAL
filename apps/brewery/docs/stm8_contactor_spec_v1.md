# STM8 v1: управление контактором (safety)

Документ описывает поведение STM8S003, управляющего **контактором силовой части**. STM8 является независимым уровнем защиты и не зависит от UI/логики рецепта.

## Роль STM8

- Единственный силовой выход STM8 — **контактор**, размыкающий питание силовой части (ТЭН через SSR и помпы через реле находятся “за контактором”).
- STM8 **не** управляет SSR и помпами напрямую.
- Логика STM8 должна быть детерминированной, минимальной и fail-safe.

## Входы STM8 (v1)

### 1) Кадр состояния STM32 → STM8 (USART RX)

- Направление: **только** STM32 → STM8.
- Кадр содержит данные состояния (как минимум температуру) + CRC.
- Сам факт приёма валидного кадра является “дыхалкой” STM32.

Решения, которые нужно зафиксировать отдельной спекой кадра:

- период кадра;
- структура полей и единицы (температура `x2`, статус уровня и т.д.);
- CRC алгоритм;
- политика N CRC подряд.

### 2) Датчик уровня (дискрет “OK/не OK”)

- Сигнал “OK” поступает на STM8 по отдельной линии (оптопара).
- Сигнал уровня является независимым условием разрешения силовой части.

### 3) Аппаратный watchdog STM8 (IWDG)

- Пропуск кормления IWDG приводит к аппаратному reset.
- После reset выход контактора должен оказаться в состоянии **разомкнут** (safe).

## Выходы STM8 (v1)

### Контактор

- `contactor = ON` разрешает питание силовой части.
- `contactor = OFF` размыкает питание силовой части.

Требование: состояние по умолчанию после старта/сброса — `OFF`.

### Дыхалка STM8 → STM32 (GPIO)

- STM8 выдаёт “дыхалку” на STM32 с детерминированным периодом.
- Если STM32 не видит дыхалку — считает STM8 неисправным и уходит в safe (выключает SSR/помпы со своей стороны).

### Статусный регистр STM8 → STM32 (GPIO)

- STM8 хранит причину последнего отключения контактора в “status register”.
- STM32 считывает статус при восстановлении связи **до** возврата управления пользователю.

## Параметры (константы прошивки STM8 v1)

- `SAFETY_TEMP_LIMIT_X2` = 204 (102.0 °C). Не конфигурируется через кадр.
- `FRAME_TIMEOUT_MS` ≈ 1000 мс (точное значение зафиксировать).
- `CRC_FAIL_LIMIT` = N подряд (точное значение зафиксировать).
- Период основного цикла STM8: 50–100 мс (зафиксировать).

## Состояния и правила управления контактором

### Определения

- **FrameValid**: получен и проверен CRC валидного кадра.
- **FrameFresh**: с момента последнего `FrameValid` прошло < `FRAME_TIMEOUT_MS`.
- **LevelOK**: вход датчика уровня = OK.
- **TempOK**: температура из последнего валидного кадра < `SAFETY_TEMP_LIMIT_X2`.
- **MCUHealthy**: STM8 исполняет основной цикл и кормит IWDG.

### Правило разрешения контактора (gating)

Контактор может быть включён только если одновременно истинны:

`FrameFresh && LevelOK && TempOK`

Если любое условие ложно — контактор должен быть выключен.

### Латчинг (ручное восстановление)

В v1 контактор отключается **с фиксацией причины** и требует ручного восстановления.

Предлагаемая политика v1:

- Любое отключение контактора (кроме “ещё не было ни одного валидного кадра”) выставляет `trip_latched=true`.
- При `trip_latched=true` контактор остаётся `OFF` даже если условия снова стали истинны.
- Сброс `trip_latched` происходит только:
  - по аппаратному Reset STM8 (кнопка/питание), или
  - (опционально) по отдельному входу “Reset” (если будет аппаратно), но **не** через USART.

Это гарантирует “без автоповтора” и предотвращает дребезг силовой части при нестабильных условиях.

## Причины отключения (коды)

STM8 фиксирует последнюю причину в статусном регистре.

Предлагаемые коды v1 (зафиксировать):

- `0x00` — нет причины / не установлено
- `0x01` — timeout кадра (нет `FrameFresh`)
- `0x02` — CRC ошибки подряд (достигнут `CRC_FAIL_LIMIT`)
- `0x03` — уровень не OK (`!LevelOK`)
- `0x04` — превышение температуры (`!TempOK`)
- `0x05` — reset по watchdog (если можно детектировать по флагам reset-cause)

Правило приоритета: если одновременно несколько условий, фиксировать наиболее “жёсткую” причину (например: temp > level > timeout > crc).

## Псевдологика основного цикла (v1)

1) Обновить “reset cause” (если доступно) и при необходимости записать `0x05`.
2) Проверить, есть ли новый кадр (`FrameReady`), если да:
   - проверить CRC; при успехе сохранить поля (температура и т.д.), сбросить счётчик CRC ошибок, обновить `last_frame_ms`;
   - при ошибке CRC увеличить счётчик; если достигли `CRC_FAIL_LIMIT` → trip.
3) Считать `LevelOK`.
4) Вычислить `FrameFresh`, `TempOK`.
5) Если `trip_latched` или любое из условий не выполнено → `contactor=OFF` + установить код причины (если ещё не установлен для текущего trip).
6) Если `!trip_latched` и все условия истинны → `contactor=ON`.
7) Обновить дыхалку STM8.
8) Кормить IWDG.

## Тестовые сценарии (минимум)

- Нет валидных кадров при старте → контактор OFF.
- Валидные кадры идут, LevelOK, TempOK → контактор ON.
- Пропали кадры > timeout → контактор OFF, причина TIMEOUT, latch.
- CRC-ошибки N подряд → контактор OFF, причина CRC, latch.
- LevelOK стал false → контактор OFF, причина LEVEL, latch.
- Температура >= 102°C → контактор OFF, причина TEMP, latch.
